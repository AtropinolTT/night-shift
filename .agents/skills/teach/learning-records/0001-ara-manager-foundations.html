# Learning Record 0001: ARA Manager Foundations ## What Was Learned The foundational workflow for `/ara-manager`: inspecting project state and logging new findings. The four ARA layers (logic/trace/staging/evidence) and their mutability rules. Five "inspect" commands (status, briefing, tree) and five "record" commands (log decision/experiment/dead_end/pivot/question, add-claim). ## Key Insights - ARA is only valuable if **current** — the two-command habit (briefing at start, log at end) is the critical practice - The lifecycle: discovery → log/stage → crystallize → advance-claim - Provenance is conservative: default to `ai-suggested` when uncertain - The staging buffer is meant for triage — not everything needs immediate crystallization ## Why Important Ties directly to mission (keep ARA updated + integrate into workflow). Without these foundation skills, the ARA project drifts stale and becomes misleading rather than useful. ## Open Questions - How to batch-crystallize when observations pile up? → Next lesson: crystallize & advance - How to auto-extract from docs like handoffs? → Later lesson: update & compile

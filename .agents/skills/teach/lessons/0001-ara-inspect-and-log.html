Inspect & Log

The fundamental cycle: know where you are, record what you discover. Every ARA workflow starts here.

Why This Matters

Your ARA project at ./ara/ already captures 7+ claims, 50+ exploration nodes, and a full experiment log for MARLIN. But research is ongoing — R11 HPO is running, the physicochemical predictor is evolving, and new findings happen every session.

An ARA project is only as valuable as it is current. If you never log new findings, the claims go stale, staged observations pile up, and the briefing becomes misleading. The two skills you'll learn here — inspecting (what's the state?) and logging (how do I record?) — are the foundation for keeping your ARA alive.

The Four Layers (Quick Refresher)

You already have all four in ./ara/:

LayerMutabilityPurposeExample
logic/ Mutable (rewritten) Current best understanding: claims, concepts, experiments, problem claims.md — C01 through C08
trace/ Append-only Journey: decisions, experiments, dead-ends, pivots exploration_tree.yaml — N01 through N50+
staging/ Append-only Buffer for observations before crystallization observations.yaml — O001 through O1xx
evidence/ Append-only Raw proof: figures, tables, README index figures/, tables/
Rule of thumb: You write to staging/ and trace/ frequently. logic/ only changes when a staged observation crystallizes into a formal claim.

Command 1: status — The Dashboard

Run this first, every session. It tells you what's happening at a glance.

/ara-manager status
ARA Project: MARLIN: Manifold-constrained Attention Residuals for LNP Interpretable Network
Domain:      Computational Biology / Drug Discovery — Molecular Property Prediction
Location:    ./ara/

Claims (N total):
  supported     3   ███
  testing       2   ██
  hypothesis    1   █
  withdrawn     1   █

Staged observations: 12  (2 stale ≥ 3 days)
Exploration nodes:   53  (18 decisions, 12 experiments, 4 dead-ends, 8 pivots, 11 questions)
Open threads:        3   (see latest session record)
Last session:        2026-06-24  (turns: 47)

Key information at a glance:

  • Claim statuses — how many are supported vs testing vs hypothesis. Are we making progress?
  • Staged observations — knowledge waiting to be formalized. "12 staged, 2 stale" means a cleanup opportunity.
  • Exploration nodes — the shape of your research: more experiments than dead-ends is healthy.
  • Open threads — dangling conversations from the last session.

Add --json for machine-readable output (useful if another tool wants to consume it):

/ara-manager status --json

Command 2: briefing — The Full Picture

Where status is the dashboard, briefing is the sitrep. Run this at the start of a session to pick up where you left off.

/ara-manager briefing

Open threads:
  #1 — R11 HPO results pending (expected ~4h)
  #2 — Physicochemical predictor R2 report needs review

Staged observations near closure:
  O114 (claim) — R10 achieves 0.6581 ROC-AUC → signal: resolution (experiment complete)
  O115 (claim) — XAI demo passes 448/468 rows → signal: affirmation (user confirmed)

Stale observations (≥3 days):
  O089 (heuristic) — "2-stage screening draft" [5 days] → needs triage

Claims: 3 supported, 2 testing, 1 hypothesis, 1 withdrawn, 0 refuted
Pre-session ritual: Open Claude Code → /ara-manager briefing → know exactly where you are. Costs < 1 second.

Command 3: tree — The Exploration History

Your ARA project tracks every decision, experiment, dead-end, pivot, and question in a DAG. Use tree to navigate it.

/ara-manager tree
N01  [decision] Adopt 7-phase migration plan
N07  [decision] Standardize on PyTorch Lightning
├─ N08  [experiment] Try big-bang rewrite
│  └─ × N09  [dead_end] Big-bang fails — too many entangled deps
N26  [question] Does Z-score normalization improve organ-level ranking?
├─ N27  [experiment] Compare raw vs Z-score normalized targets
│  └─ N28  [decision] Adopt ORGAN_STATS normalization
...

Useful flags:

  • --node N12 — show only the subtree rooted at N12
  • --type experiment — filter to experiments only
  • --depth 2 — limit depth (useful for a broad tree)
When to use tree: When you need to understand why a decision was made, or find the experiment that verified a claim. The tree is your research provenance graph.

The Core Skill: log — Recording What Happens

This is THE command you'll use most. When something happens in research — you make a choice, run an experiment, hit a wall, change direction, or have a question — log it immediately.

log decision "description"
A deliberate choice between alternatives. Always note what you chose and what you rejected.
/ara-manager log decision "Use XGBoost for physicochemical predictor primary model" --provenance user
[ara-manager] Logged N54 (decision) — "Use XGBoost for physicochemical predictor primary model"
log experiment "description"
An experiment you ran. Include the outcome, even if it's inconclusive — negative results are valuable.
/ara-manager log experiment "R11 HPO: LR=0.0005, growth_rate search [0.003,0.025]" --provenance ai-executed
[ara-manager] Logged N55 (experiment) — "R11 HPO launched on gpu1"
log dead_end "description"
A path that didn't work. Record why it failed and what you learned — this prevents repeating mistakes.
/ara-manager log dead_end "PDI prediction R²≈0 — NaN-imputed zeros drove illusion" --provenance user
[ara-manager] Logged N56 (dead_end) — "PDI prediction fails — artifact mirage"
log pivot "description"
A change in direction — from → to, and what triggered it.
/ara-manager log pivot "Switched from RF to XGBoost for feature selection" --provenance user
[ara-manager] Logged N57 (pivot) — "RF → XGBoost for feature selector"
log question "description"
An open research question. Questions stay in the tree until they're answered by an experiment or decision.
/ara-manager log question "Does ensemble uncertainty improve active learning sampling?" --provenance user
[ara-manager] Logged N58 (question) — "Ensemble uncertainty for active learning?"
Provenance rules (just two to remember):
  • user — you made the call or said it
  • ai-executed — Claude ran the experiment/generated the result
  • When unsure? Use ai-suggested (the safe default)

Bonus: add-claim — Staging a New Claim

When you realize something is true but it hasn't been formally claimed yet, stage it as an observation. It'll sit in staging/observations.yaml until it crystallizes.

/ara-manager add-claim "Physicochemical predictor EE model achieves R²=0.93 on 5-fold CV" --provenance user --bound-to N30,N45
[ara-manager] Staged O123 (claim) — "Physicochemical predictor EE model achieves R²=0.93 on 5-fold CV"
  Currently staged. Run `/ara-manager crystallize O123 --via` when ready to promote.
The skill checks if your claim reads as a falsifiable assertion. If you write "we should try X" it'll warn you — that's a decision, not a claim.

The Lifecycle (3-Minute Overview)

Here's how knowledge flows through your ARA project:

🔬 Discovery → 📝 log (trace) or 📋 add-claim (staging) → 💎 crystallize (logic) → 📈 advance-claim
  1. Discovery — you run an experiment, make a decision, or realize something
  2. Stage or Trace — log goes directly to the exploration tree; add-claim goes to staging buffer
  3. Crystallize — when you're confident, promote the staged observation into logic/claims.md
  4. Advance — as evidence accumulates, move the claim through hypothesis → testing → supported

You don't need to crystallize everything today. The staging buffer is meant to hold observations for later triage.

🔧 Your Turn: 5-Minute Practice

Exercise 1 — Check your current state

Run both commands and observe the output:

! /ara-manager status
! /ara-manager tree --type question --depth 1

What open questions exist? Are any stale observations waiting for you?

Exercise 2 — Log something real

Think of one thing that happened in your last research session. A decision you made, an experiment you ran, or a question that came up. Log it:

! /ara-manager log decision "<what you decided>" --provenance user

Don't overthink it — just get it recorded.

Exercise 3 — Stage an observation

Is there a finding you're confident about but haven't claimed yet? Stage it:

! /ara-manager add-claim "<your falsifiable assertion>" --provenance user

Foundation

What is an ARA?

An Agent-Native Research Artifact (ARA) organizes a research project into four interlocking layers that AI agents can navigate, reproduce, and extend.

logic/

What & Why — Mutable current understanding

  • problem.md Observations → gaps → insight
  • claims.md Falsifiable claims (C01, C02, …)
  • concepts.md Formal term definitions
  • experiments.md Declarative experiment plans
  • related_work.md Typed dependency graph
  • solution/ Constraints & heuristics

src/

How — Concrete artifacts

  • environment.md Reproducibility (deps, hardware)
  • configs/ Configuration files
  • execution/ Run scripts & pipelines
  • data/ Datasets & processed results

trace/

Journey — Append-only

  • exploration_tree.yaml Research DAG
  • pm_reasoning_log.yaml Organizational decisions
  • sessions/ Per-day session records

staging/

Buffer — Append-only

  • observations.yaml Raw observations awaiting closure

evidence/

Raw Proof — Append-only

  • tables/ & figures/ & proofs/
  • Each file maps back to specific claims

Rule of thumb: logic/ rewrites in place (current best understanding); everything else only appends (the immutable record).

Hands-on

Step-by-Step Walkthrough

Follow along to create and manage an ARA research project from scratch.

Initialize the Project

Create a new ARA project with a title, domain, and keywords. This scaffolds the full directory tree and seed files.

Terminal
$ /ara-manager init "Protein-Ligand Binding with GNNs" \
  --domain "computational chemistry" \
  --keywords gnns,drug-discovery,binding-affinity
[ara-manager] Initialized ARA project at ./ara/.
Files: 13
Next: /ara-manager add-claim "..." or /ara-manager log decision "..."

Check Project Status

Get a compact dashboard showing claim counts, staged observations, exploration nodes, and the last session.

Terminal
$ /ara-manager status
ARA Project: Protein-Ligand Binding with GNNs
Domain: computational chemistry
Location: ./ara/
Claims (0 total): (none)
Staged observations: 0
Exploration nodes: 0
Last session: (none)

Log a Research Decision

Record a decision directly to the exploration tree (append-only). The provenance defaults to user when quoted.

Terminal
$ /ara-manager log decision "Chose AdamW over SGD for optimizer" \
  --alternatives "SGD,RMSProp" --evidence "C01"
[ara-manager] Logged N01 (decision) — "Chose AdamW over SGD"

Stage a Falsifiable Claim

All observations start in the staging buffer. Claims are validated for falsifiability — warnings appear but don’t block.

Terminal
$ /ara-manager add-claim \
  "Our GNN achieves >0.85 ROC-AUC on ChEMBL binding-affinity benchmark" \
  --tags gnns,chembl,perf
[ara-manager] Staged O01 (claim) — pending crystallization

Log an Experiment

Record experimental results. The agent derives result and hypothesis fields from your description.

Terminal
$ /ara-manager log experiment \
  "Trained 3-layer GNN on ChEMBL — 0.87 ROC-AUC, 0.52 RMSD"
[ara-manager] Logged N02 (experiment) — "3-layer GNN on ChEMBL"

Crystallize the Claim

When evidence arrives, promote the staged observation with a closure signal. Here, the experiment resolved in its favor.

Terminal
$ /ara-manager crystallize O01 --via resolution
[ara-manager] Crystallized O01 → C01 (claim) in logic/claims.md via resolution
# The observation is marked promoted:true but NOT deleted.
# The staging trail is part of the permanent record.

Advance the Claim

Update the claim’s status from hypothesis to supported now that experimental evidence confirms it.

Terminal
$ /ara-manager advance-claim C01 supported --note "N02 confirms: 0.87 ROC-AUC > 0.85 threshold"
[ara-manager] Advanced C01: hypothesis → supported

Run Validation

Seal Level 1 checks all 17 structural constraints — file presence, cross-layer references, provenance, tree hygiene, and self-consistency.

Terminal
$ /ara-manager validate
[validate] Seal Level 1: PASS (17/17 checks)

Organize Daily Maintenance

The multi-agent maintenance loop handles scouting, extraction, review, execution, and verification automatically.

Terminal
$ /ara-manager organize daily
[ara-manager] organize daily — running L1 maintenance.
Report will be written to ara/daily/2026-06-26.md.
Guidelines

Best Practices

These rules ensure your ARA project remains coherent, traceable, and machine-readable.

①

Stage Before Crystallizing

Always add observations to the staging buffer first. Crystallize only when a closure signal fires. This preserves the evidence trail and prevents premature commitment.

// Preferred flow:
/ara-manager add-claim "GNN achieves >0.85 AUC"
/ara-manager crystallize O01 --via affirmation
②

Write Falsifiable Claims

A good claim names a measurable outcome with a scope qualifier. Vague claims can’t be tested or refuted.

✓ "Model achieves >90% accuracy on dataset X"
✗ "The model works well"
✗ "We should try a different approach"
③

Honor Layer Mutability

logic/ is mutable — edit in place. trace/, staging/, and evidence/ are append-only — never delete or rewrite prior entries. This separation preserves the journey record.

✓ Append to staging/observations.yaml
✗ Delete a staged observation
④

One-Step Transitions

Advance claims incrementally: hypothesis → testing → supported. Jumping from hypothesis directly to supported requires both experimental results and explicit affirmation.

✓ hypothesis → testing → supported
✗ hypothesis → supported (no evidence)
⑤

Never Invent IDs

Always read the target file first to find the highest existing ID, then increment. Guessing IDs creates duplicates and breaks cross-layer references.

✓ Read highest C{XX}, allocate C{XX+1}
✗ Guess C02 without reading claims.md
⑥

Default to Non-Promotion

Crystallization is the exception, not the default. An observation stays staged until a clear closure signal fires. When in doubt, keep it staged.

✓ "Let me check the results first" = keep staged
✓ "I confirmed this works" = crystallize via affirmation
⑦

Cross-Layer Consistency

After every write, verify that cross-layer bindings stay valid. Claim Proof: fields must reference real experiment IDs; PAPER.md claims_summary must match the claim count.

✓ C01 Proof: [E05] — E05 exists in experiments.md
✗ C01 Proof: [E99] — E99 doesn't exist
⑧

Provenance is Conservative

Default to ai-suggested when uncertain. Never upgrade to user-revised without an explicit user revision this turn. The provenance track is your trust anchor.

✓ Default: ai-suggested (safe)
✓ User said "yes, that's right" = user
✗ Assume user-revised without evidence
Trust

Provenance & Claim Transitions

The ARA standard defines four provenance levels and a strict state machine for claim statuses.

Provenance Values

ValueUse When
userThe user made the statement in first person or quoted verbatim. Highest trust.
ai-suggestedThe AI proposed it; the user hasn’t yet confirmed. Safe default.
ai-executedThe AI performed the action — ran an experiment, executed a query, generated a result.
user-revisedThe AI proposed something, the user edited it. Hybrid provenance — never inferred automatically.

Allowed Claim Status Transitions

hypothesis → testing → supported ↑ weakened | refuted | withdrawn | revised

revised is a transition marker — must settle to testing or hypothesis next.
Terminal states (refuted, withdrawn) can only revive through revised.

Ecosystem

The ARA Skill Family

ARA Manager is one of five companion skills that together form a complete research knowledge management system.

/ara-manager
On-demand project operations — add claims, log events, check status, run validation. The command hub you’re learning here.
/compiler
One-shot ARA construction from papers, code repos, logs, and notes. Runs the full 4-stage epistemic protocol.
/research-manager
Per-turn epilogue that records research events with correct provenance. Runs as a background process during active research.
/rigor-reviewer
Seal Level 2 deep epistemic review across six dimensions: evidence relevance, falsifiability, scope calibration, argument coherence, exploration integrity, methodological rigor.
/improve-codebase-architecture
Analyzes src/ for deepening opportunities — surface shallow modules, find missing seams, propose refactors.
